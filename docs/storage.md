# Charon: Storage Design

## The Core Data Structure

Every response is a node in a singly-linked list:

```
resp_A { prev: null,    input: [...], output: [...] }
   ↑
resp_B { prev: resp_A,  input: [...], output: [...] }
   ↑
resp_C { prev: resp_B,  input: [...], output: [...] }
```

The access patterns that drive the storage design are:

| Operation | Frequency | Description |
|-----------|-----------|-------------|
| **Store** | Every request | Write one new response record |
| **Point lookup** | GET /responses/{id}, every chain walk step | Fetch one record by primary key |
| **Chain walk** | Every turn with `previous_response_id` | Walk back to the chain root, flatten into context |
| **List** | GET /responses (pagination) | List responses by model/user/time |

There are no range scans over content. No joins. The hot path is: PK lookup → chain walk → flatten.

## Storage Interface

The storage layer is split into two interfaces:

```go
// ListOptions controls filtering and pagination for List calls.
// Zero values mean "no filter" / "use default".
type ListOptions struct {
    Owner  string // filter by owner principal; empty = no filter (Phase 1: always empty)
    Cursor string // pagination cursor; empty = start from beginning
    Limit  int    // max results per page; 0 = use implementation default
    // Phase 2+: Model, Status, CreatedAfter, CreatedBefore, ChainRootID
}

// IndexStore manages chain linkage and lightweight metadata.
// Phase 1: PK operations only. Chain traversal lives in the service layer (BuildContext),
// which walks backward via serial Get calls — fast enough with SQLite's in-memory page cache.
type IndexStore interface {
    Put(ctx context.Context, meta ResponseMeta) error
    Get(ctx context.Context, id string) (ResponseMeta, error)
    Delete(ctx context.Context, id string) error
    List(ctx context.Context, opts ListOptions) ([]ResponseMeta, error)
}

// Phase 2 additions (PostgreSQL): replace N serial Get round-trips with indexed range scans.
// GetChainRange(ctx, chainRootID, fromPos, toPos) fetches records in [fromPos, toPos] order.
// GetChain(ctx, chainRootID) is shorthand for GetChainRange(ctx, id, 0, MaxInt).
// Both are non-breaking additions; SQLite implements them via the existing (chain_root_id, position) index.

// PayloadStore manages item content (input/output items).
// Get returns ErrNotFound if the key does not exist — use it instead of a separate Exists check.
type PayloadStore interface {
    Put(ctx context.Context, key string, data []byte) error
    Get(ctx context.Context, key string) ([]byte, error)
    Delete(ctx context.Context, key string) error
}
```

`ResponseMeta` is the lightweight index record:

```go
type ResponseMeta struct {
    ID                 string
    PreviousResponseID *string
    ChainRootID        string    // denormalized at write time
    Position           int       // 0-based ordinal in chain, denormalized at write time
    IsCheckpoint       bool
    PayloadKey         string    // path/key to this turn's delta (input+output items)
    CheckpointKey      *string   // path/key to the full flat_context blob; non-nil only when IsCheckpoint=true
    OwnerPrincipal     string
    AccessAttributes   map[string][]string
    Model              string
    Status             string
    CreatedAt          int64
    ExpiresAt          *int64
}
```

`ChainRootID` and `Position` are denormalized at write time. They eliminate the need for recursive chain traversal during reads: a full chain is a single `WHERE chain_root_id = ? ORDER BY position` scan rather than N serial PK lookups.

## Payload Schema

Each response's payload is the JSON-serialized content of items that must be preserved for chain reconstruction. This is separate from the lightweight metadata above.

**Minimum required content:**

```go
type ResponsePayload struct {
    ID                 string   // redundant with metadata but self-describing
    PreviousResponseID *string
    InputItems         []Item   // what the caller sent this turn
    OutputItems        []Item   // what the model produced this turn
}
```

`Item` is a discriminated union covering:

| Type | Description |
|------|-------------|
| `message` | User/assistant/system text |
| `function_call` | Tool call generated by the model |
| `function_call_output` | Tool result provided by the caller |
| `reasoning` | Opaque reasoning blob with `encrypted_content` |
| `compaction` | Opaque compaction blob with `encrypted_content` |

`encrypted_content` fields on `reasoning` and `compaction` items are opaque provider blobs. They must be stored verbatim and returned verbatim. They cannot be inspected, modified, or reconstructed.

**Fields stored for API completeness but NOT used for reconstruction:**
`status`, `created_at`, `model`, `usage`, `error`, `temperature`, `top_p`, `tools`, `tool_choice`, `instructions`, `metadata`, `service_tier`.

`instructions` is explicitly excluded from the reconstructed context — it must be re-supplied by the caller on each turn.

## Storage Backend Options

### Option A — Pure SQL (Baseline)

Both metadata and payload content live in a single SQL table.

**Schema:**

```sql
CREATE TABLE responses (
    id                   TEXT PRIMARY KEY,
    previous_response_id TEXT REFERENCES responses(id),
    chain_root_id        TEXT NOT NULL,
    position             INTEGER NOT NULL,
    is_checkpoint        BOOLEAN NOT NULL DEFAULT FALSE,
    owner_principal      TEXT,
    model                TEXT,
    status               TEXT,
    created_at           INTEGER NOT NULL,
    expires_at           INTEGER,
    payload              JSONB,           -- full input+output items
    checkpoint_payload   JSONB            -- non-null only on checkpoint rows
);

CREATE INDEX idx_responses_chain ON responses(chain_root_id, position);
CREATE INDEX idx_responses_owner ON responses(owner_principal, created_at);
```

**Chain retrieval:**

```sql
SELECT * FROM responses
WHERE chain_root_id = ?
ORDER BY position ASC;
```

**Complexity:**

| Operation | Time | Storage |
|-----------|------|---------|
| Store | O(1) | O(payload size) per row |
| Point lookup | O(1) PK | — |
| Full chain | O(N) sequential scan | — |

**Pros:** Single system; full ACID; trivial to deploy; arbitrary queries (by user, model, time). The recursive CTE (`WITH RECURSIVE`) is available as a fallback if `chain_root_id` denormalization is absent, but the index makes it unnecessary.

**Cons:** Large payload columns bloat the table and hurt index performance. No compression across entries. For PostgreSQL, large JSONB columns cause TOAST churn. For SQLite, single writer bottleneck.

**Best fit:** Single-node deployment, low volume, operational simplicity is paramount.

---

### Option B — SQL Index + Object Store Payloads (Recommended General Architecture)

SQL holds lightweight metadata only. Payload content lives in object storage.

**SQL schema:**

```sql
CREATE TABLE responses (
    id                   TEXT PRIMARY KEY,
    previous_response_id TEXT,
    chain_root_id        TEXT NOT NULL,
    position             INTEGER NOT NULL,
    is_checkpoint        BOOLEAN NOT NULL DEFAULT FALSE,
    owner_principal      TEXT,
    model                TEXT,
    status               TEXT,
    created_at           INTEGER NOT NULL,
    expires_at           INTEGER,
    payload_key          TEXT NOT NULL,       -- path in object store
    payload_size_bytes   INTEGER
);
```

**Object store key pattern:**

```
{chain_root_id}/{position:08d}_{response_id}.json[.zst]
```

Compression (zstd) is optional and configurable. JSON conversation payloads compress to ~25–30% of original size (3–4× reduction) with negligible CPU overhead at typical payload sizes (1–50 KB). Enable it when storage cost is measurable; leave it off when simplicity is preferred. The `.zst` extension signals compressed payloads; uncompressed payloads omit it. Both forms must be readable by the PayloadStore implementation.

Using `position` in the key means: (1) keys sort correctly, and (2) an object store prefix scan on `{chain_root_id}/` independently enumerates the chain without SQL.

**Full chain retrieval:**

```sql
SELECT payload_key FROM responses
WHERE chain_root_id = ?
ORDER BY position ASC;
-- Then parallel GET all payload_key values from object store.
```

The parallel GETs make full history retrieval effectively O(max_single_object_latency) in wall time, not O(N).

**Complexity:**

| Operation | Time | Storage |
|-----------|------|---------|
| Store | O(1) SQL + O(1) object PUT | O(~200 bytes) SQL row + O(payload) compressed object |
| Point lookup | O(1) PK + O(1) GET | — |
| Full chain | O(1) SQL scan + O(N) parallel GETs | — |

**Pros:** SQL table stays narrow and fast regardless of payload size. Object store handles compression natively (zstd, 60–70% reduction on JSON). Prefix scan on object store provides a backup retrieval path independent of SQL. Scales horizontally.

**Cons:** Two failure domains — no atomic transaction spans SQL and object store. Object store latency (even local MinIO) is higher than a DB row read for single-entry access.

**Best fit:** General-purpose production deployment.

### Option C — Snapshot-Augmented Chain

For conversations exceeding ~100 turns, chain traversal latency becomes measurable even with parallel fetches. Periodic materialized snapshots bound the worst case.

**Additional table:**

```sql
CREATE TABLE snapshots (
    snapshot_id        TEXT PRIMARY KEY,
    chain_root_id      TEXT NOT NULL,
    snapshot_at_pos    INTEGER NOT NULL,   -- position of last response included
    up_to_response_id  TEXT NOT NULL,
    storage_key        TEXT NOT NULL,      -- object store path for snapshot blob
    entry_count        INTEGER NOT NULL,
    created_at         INTEGER NOT NULL
);

CREATE INDEX idx_snapshots_chain ON snapshots(chain_root_id, snapshot_at_pos DESC);
```

**Snapshot blob:** the complete flattened `[]Item` array for positions 0..K, serialized and compressed.

**Retrieval algorithm given response at position P:**

1. Find the most recent snapshot where `chain_root_id` matches and `snapshot_at_pos ≤ P`.
2. Fetch the snapshot blob (positions 0..S).
3. Fetch individual payloads for positions S+1..P from the `responses` table (the "tail").
4. Concatenate snapshot items + tail items.

**Worst case:** at most K individual payload fetches + 1 snapshot fetch, regardless of total chain length N.

**Checkpoint write trigger (inside the store path):**

```go
func (s *Store) SaveResponse(ctx context.Context, rec *ResponseRecord, fullContext []Item) error {
    if rec.Position > 0 && rec.Position%s.CheckpointInterval == 0 {
        // fullContext is already in hand — materializing is free
        rec.IsCheckpoint = true
        rec.CheckpointPayload = serialize(fullContext, rec.OutputItems)
    }
    return s.put(ctx, rec)
}
```

**Complexity (K = checkpoint interval):**

| Operation | Time | Storage |
|-----------|------|---------|
| Store (normal) | O(1) | O(payload) |
| Store (checkpoint) | O(1) compute + O(chain_length) write | +O(K × avg_payload) per checkpoint |
| Full chain | O(K) tail fetches + O(1) snapshot | — |

**Pros:** Retrieval cost bounded at O(K) regardless of N. Snapshots also serve as recovery artifacts. Works well with object storage for snapshot blobs.

**Cons:** Snapshot records are large — proportional to total conversation length at that point. Write amplification on checkpoint turns. Snapshot invalidation complexity on deletion (GDPR). Branching chains (DAG) break snapshot reuse.

**Recommended K:** 10 for read-heavy sessions (agents that are resumed frequently), 50 for write-heavy sessions (rarely resumed). No checkpoint for chains shorter than K.

### Option D — Packed Segments with Internal Index

Inspired by Git packfiles. Multiple response payloads are grouped into a single compressed object. Reduces object store overhead (object stores are inefficient at millions of tiny files) and improves compression ratio (cross-entry compression of repeated JSON structure).

**SQL index:**

```sql
CREATE TABLE entries_index (
    response_id       TEXT PRIMARY KEY,
    chain_root_id     TEXT NOT NULL,
    global_position   INTEGER NOT NULL,
    pack_id           TEXT NOT NULL REFERENCES packs(pack_id),
    position_in_pack  INTEGER NOT NULL,
    byte_offset       INTEGER NOT NULL   -- byte offset within pack blob
);

CREATE TABLE packs (
    pack_id           TEXT PRIMARY KEY,
    chain_root_id     TEXT NOT NULL,
    pack_start_pos    INTEGER NOT NULL,
    pack_end_pos      INTEGER NOT NULL,
    entry_count       INTEGER NOT NULL,
    storage_key       TEXT NOT NULL,
    size_bytes        INTEGER NOT NULL,
    sealed            BOOLEAN NOT NULL DEFAULT FALSE
);
```

**Pack blob format (object store):**

```
[ 4-byte magic ]
[ 4-byte entry_count ]
[ entry_count × { 8-byte response_id_hash, 4-byte byte_offset } ]  ← internal index
[ entry_0_length (4 bytes) | entry_0_data ]
[ entry_1_length (4 bytes) | entry_1_data ]
...
[ compressed with zstd ]
```

The internal index lets a single entry be fetched by HTTP Range request without reading the full pack.

**Retrieval:**

- Single entry: SQL index lookup → `byte_offset` → HTTP Range GET (fetch only that entry's bytes)
- Full history: `SELECT pack_id, storage_key FROM packs WHERE chain_root_id = ? ORDER BY pack_start_pos` → ceil(N/K) parallel full GETs

**Complexity:**

| Operation | Time | Storage |
|-----------|------|---------|
| Store (into open pack) | O(1) staging | O(entry) in staging |
| Seal pack (every K entries) | O(K) compress+write | O(K × entry) → compressed |
| Single entry (sealed) | O(1) SQL + O(entry) Range GET | — |
| Full history | O(ceil(N/K)) parallel GETs | — |

**Pros:** Best storage efficiency (cross-entry compression). Full history in ceil(N/K) parallel fetches. Individual entries still addressable. Sealed blobs are immutable and trivially cacheable.

**Cons:** Most complex implementation — staging, sealing, open-pack management. Two forms of every entry exist until sealed. GDPR deletion requires rewriting packs. K tuning is sensitive.

**Best fit:** High scale, read-heavy workloads where retrieval efficiency outweighs implementation complexity.

## Strategy Comparison

| | A: SQL | B: SQL + Object | C: Snapshots | D: Pack Segments |
|---|---|---|---|---|
| Store complexity | O(1) | O(1) | O(1) amortized | O(1) amortized |
| Single entry retrieval | O(1) PK | O(1) PK + 1 GET | O(1) PK | O(1) index + Range GET |
| Full chain | O(N) serial | O(N) parallel | O(K) tail + snapshot | O(N/K) parallel |
| Storage efficiency | Low | Medium | Medium + redundancy | High |
| Operational complexity | Low | Medium | High | Very high |
| Deletion/GDPR | Trivial | Easy | Hard (invalidate snapshots) | Hard (rewrite packs) |
| Best fit | Low scale | General purpose | Long conversations | High scale, read-heavy |

**Recommendation:** Start with Option B (SQLite IndexStore + filesystem PayloadStore) from day one. The IndexStore/PayloadStore interface split is the target architecture — starting with Option A (payloads inline in SQL) means implementing the split twice. Option A's operational simplicity advantage is largely preserved: single binary, single data directory. Add Option C snapshotting (checkpoints) from Phase 1 to bound chain-walk cost; they are co-located with chain payloads in the filesystem layout. Migrate to Postgres + object store (Option B backend swap) when SQLite write serialization or storage size becomes measurable.

## Consistency: The Two-Failure-Domain Problem

Options B, C, and D split state across SQL and an object store. There is no atomic transaction that spans both. Every write creates a window for divergence.

### Failure modes

**SQL write succeeds, object PUT fails** — The index row points to a key that does not exist. Any read of this entry hits a 404. This is the **worse** failure: it silently corrupts the chain. The read path must treat a missing object as a hard error, not a soft miss, and surface `chain_corrupted` rather than returning partial history.

**Object PUT succeeds, SQL write fails** — The blob exists with no index entry. It is invisible and unreachable, leaking storage. This is the **safer** failure: no corrupt reads, just a ghost object. Detectable by periodic reconciliation.

### Write ordering cannot solve this

Writing the object first prevents broken pointers but worsens ghost objects. Writing SQL first prevents ghost objects but worsens broken pointers. Neither ordering is "safe" — you trade one failure mode for the other.

### Mitigations

**Write-intent log (two-phase commit in SQL):**

```sql
CREATE TABLE write_intents (
    intent_id      TEXT PRIMARY KEY,
    response_id    TEXT NOT NULL,       -- canonical ID from inference server; reservation_id as fallback if inference failed before canonical known
    reservation_id TEXT,                -- rsrv_... from the preceding resolve call; NULL for new chains
    payload_key    TEXT NOT NULL,
    phase          TEXT NOT NULL,       -- 'pending' | 'file_written' | 'committed' | 'failed'
                                        -- future streaming phases: 'stream_open' | 'committed' | 'failed'
    created_at     INTEGER NOT NULL,
    updated_at     INTEGER NOT NULL     -- set equal to created_at on insert; updated on each phase transition
);
```

`updated_at` is present from day one to avoid a schema migration when the recovery worker logic is refined. On INSERT, `created_at` and `updated_at` are set to the same value. Each phase transition updates `updated_at`.

Recovery condition: `phase IN ('pending', 'file_written') AND updated_at < (now - stale_threshold)`. Default stale threshold: 5 minutes (configurable via `storage.write_intent_stale_threshold_ms`). Generous enough that no legitimate in-flight write would still be in a non-terminal phase; short enough to recover promptly after a crash.

Write sequence (triggered by the store call — `POST /responses/{canonical_id}`):
1. INSERT `write_intents` row with `phase = 'pending'`, `response_id = canonical_id`, `reservation_id` (if continuation)
2. PUT object (idempotent: key is deterministic, derived from `canonical_id`)
3. UPDATE `write_intents` to `phase = 'object_written'`
4. INSERT `responses` row
5. UPDATE `write_intents` to `phase = 'committed'`

No write-intent exists before the store call. If the proxy never calls store (e.g., `store: false`, inference failure without a failure notification), there is no orphaned state to recover.

**Chunked streaming write sequence (post-Phase-1):**
1. On stream open: INSERT `write_intents` with `phase = 'stream_open'`
2. Per chunk: accumulate items in Charon (in-memory or temp staging)
3. On stream close (final chunk received): PUT object, INSERT `responses` row, UPDATE `write_intents` to `phase = 'committed'`

Recovery for stale `stream_open` intents: same stale threshold applies. The recovery worker marks them `failed`. Partial chunks written to staging are discarded. The `reservation_id` field allows correlating orphaned streams to their originating resolve calls in logs.

On restart, a recovery process finds rows in `pending` or `object_written` and either completes them (re-PUT is safe because the key is deterministic) or marks them `failed`.

**Idempotent, deterministic keys:** key the object store entry by `response_id`. Re-PUT after a crash writes the same bytes to the same key — safe to retry unconditionally.

**Hardened read path:** treat a missing object (404) as `ErrChainCorrupted`. Surface it explicitly rather than returning silent partial history.

**Background reconciliation:** periodic job that:
- Finds stale `write_intents` in `pending`/`object_written` → complete or mark failed
- Scans object store keys with no SQL row → log as leaked objects
- Spot-checks SQL payload_key values → verify objects exist

### When to avoid the two-domain problem entirely

If true atomicity is required (compliance logging, audit trails), accept the performance trade-off of Option A (pure SQL, single failure domain). The split-domain design is a performance optimization — pay for the complexity only when the gain is measurable.

## Hot Path Caching (Phase 2)

Deferred to Phase 2. With SQLite + local filesystem + checkpoints (K=10), the Phase 1 hot path is: 1 PK lookup + at most K filesystem reads — fast enough that a cache adds complexity without measurable benefit.

When introduced, the cache belongs in **Charon** (not the proxy), so all proxy instances benefit from the same warm cache. The proxy may independently add its own connection-local cache as a separate optimization.

The high-value cache: `response_id → flat_context []Item`, keyed by the head response ID. A cache hit on turn N+1 eliminates all storage reads for that resolve call. Cache at the record level (`id → ResponseMeta`) is a secondary optimization for PostgreSQL round-trip absorption.

Proxy-side caching (e.g., connection-local cache for WebSocket sessions) is a separate concern and can be added independently.

---

## TTL and Deletion

Stored responses expire after 30 days by default (matching OpenAI's behavior for `store: true` responses). Responses stored with `store: false` are never written to durable storage — they live only in a per-connection in-memory cache for the duration of the WebSocket session.

TTL enforcement via a background worker scanning `expires_at < now`. The worker runs at startup and on a configurable interval (default: hourly).

Deletion is a point operation: `DELETE /responses/{id}` removes that response only, with no effect on other responses in the chain. This matches the spec — "deleting a response has no effect on other responses."

A resolve walk that encounters a missing node returns `previous_response_not_found`. The caller is responsible for not chaining from a deleted response. Charon does not refuse deletion of non-leaf nodes and does not cascade.

Deletion and GDPR: Options A and B support row/object deletion in O(1). Options C and D require snapshot invalidation (C) or pack rewriting (D). For compliance use cases, prefer Option A or B.

## Checkpoint Implementation Detail

The checkpoint optimization converts O(N) chain traversal to O(K) by materializing the full flat context every K turns.

**Reconstruction with checkpoints:**

```go
func (s *Store) BuildContext(ctx context.Context, prevID string) ([]Item, error) {
    var delta []*ResponseRecord
    id := &prevID
    for id != nil {
        rec, err := s.Get(ctx, *id)
        if err != nil {
            return nil, err
        }
        if rec.IsCheckpoint {
            // Found materialized base: load full context from checkpoint blob.
            items, err := s.Payloads.GetCheckpoint(ctx, *rec.CheckpointKey)
            if err != nil {
                return nil, err
            }
            for i := len(delta) - 1; i >= 0; i-- {
                items = append(items, delta[i].InputItems...)
                items = append(items, delta[i].OutputItems...)
            }
            return items, nil
        }
        delta = append(delta, rec)
        id = rec.PreviousResponseID
    }
    // No checkpoint: reconstruct from root.
    slices.Reverse(delta)
    var items []Item
    for _, rec := range delta {
        items = append(items, rec.InputItems...)
        items = append(items, rec.OutputItems...)
    }
    return items, nil
}
```

**Checkpoint record sizing:** At K=10, a checkpoint at turn 100 contains 100 turns of history. For conversations with 500 tokens/turn, that is 500 × 100 = 50,000 tokens, approximately 200 KB uncompressed. With zstd: ~60–80 KB. Store checkpoint payloads in the object store, not inline in the SQL row.

**Write amplification:** At K=10, every 10th write is ~10× the size of a normal write. For read-heavy sessions (agents resumed frequently), lower K is worth the write cost. For write-heavy sessions (rarely resumed), use higher K to minimize storage.
